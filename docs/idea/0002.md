# Executive Proposal & Technical RFC: Project "Chronos Edge"
**The Omnipresent Context OS & Distributed Local RAG Memory Engine**

## 1. Executive Summary

Project Chronos Edge is an architectural expansion of the original Chronos R&D initiative. Moving beyond a "Personal Context Engine" confined to desktop screen captures, this evolution transforms Chronos into a **Distributed Semantic Vision Hub**. It captures live physical-world feeds from diverse, low-cost hardware nodes (microcontrollers, drones, IP cameras), translates these visual events into semantic logs via local AI, and enables natural language retrieval of physical events.

**The Business Differentiator:** While enterprise physical security and smart-home systems rely on expensive cloud subscriptions and compromise privacy by streaming home/factory feeds to third-party servers, Chronos Edge retains the **100% Local-First / Edge Computing architecture**. All vision processing, object detection, and semantic storage occur strictly on the user's localized base station (PC/Server).

* **For Marketing:** *"The AI that watches over your entire world—digital and physical—without a single pixel ever leaving your network."*
* **For Finance:** We pair ultra-low-cost commodity hardware ($5 microcontrollers) with zero-OpEx local AI inference, creating an infinite-ROI monitoring solution.

## 2. Product Vision & Go-to-Market

### The Problem
Traditional physical monitoring is dumb and noisy. Motion-activated cameras trigger false alarms for falling leaves or shadows, forcing users to manually scrub through hours of video timelines to find a specific event. Furthermore, streaming RTSP feeds to cloud providers for AI analysis introduces severe privacy liabilities and massive bandwidth costs.

### The Solution (User Journey)
1. **Distributed Silent Capture:** Cheap hardware nodes (e.g., ESP32-CAMs glued to a porch, a drone doing a perimeter sweep) capture frames at intervals and send them over the local WiFi network.
2. **Intelligent Translation:** The Chronos Edge base station receives the payload. The local Vision Language Model (VLM) ignores the falling leaves and logs: *"Delivery driver in a blue uniform left a package at the front door at 2:15 PM."*
3. **Instant Retrieval:** The user opens the Chronos terminal and asks: *"Did anyone approach the garage today while I was out?"* The system retrieves the exact semantic log and the corresponding captured frame.



### Future Ecosystem Use Cases
* **Smart Security (Zero False Positives):** Replacing legacy motion-detectors. The system only triggers an alert if the semantic log contains specific threats (e.g., "Person holding a tool near the window" vs. "Stray cat walking").
* **Industrial IoT & Safety:** Cameras positioned on factory floors or construction sites. Chronos passively logs: *"Worker in sector 4 is not wearing a hard hat"* or *"Forklift blocking the emergency exit."*
* **Agricultural/Asset Monitoring:** Solar-powered microcontrollers monitoring crop growth, gate statuses, or livestock movement, feeding semantic data back to a central farm server.

## 3. Detailed Technical Architecture

To guarantee feasibility, the system pivots from a single-node desktop daemon to a **Hub-and-Spoke Distributed System**. The Rust core is perfectly suited for handling highly concurrent, asynchronous network I/O.



### Component A: The Omni-Receiver (Ingestion Hub)
* **Role:** The central nervous system. It replaces the local OS screen-capture loop with an asynchronous network listener.
* **Technical Challenge:** Handling hundreds of concurrent image uploads from spotty IoT connections without blocking the main Tokio executor.
* **Solution:** An embedded, lightweight HTTP/REST or MQTT server (built with Rust's `axum` or `rumqttc`). Hardware nodes act as stateless workers, simply doing a `POST /ingest` with a binary JPEG payload and a `node_id`. The receiver places these into the existing RAM-only ring buffer.

### Component B: The Hardware Spokes (Edge Nodes)
* **Role:** The distributed "eyes."
* **Equipment Examples:**
    * **ESP32-CAM:** Ultra-cheap ($5), WiFi-enabled microcontrollers. Programmed in C++ to wake up, take a JPEG, HTTP POST it to the Chronos base station, and go back to deep sleep.
    * **RTSP IP Cameras:** Existing home security cameras. A dedicated Tokio task in Chronos pulls a frame from the RTSP stream every *n* seconds.
    * **Autonomous Drones:** Sending telemetry and frames back to the base station during a pre-programmed flight path.



### Component C: Edge Vision Worker (Local VLM)
* **Role:** The translator. 
* **Technical Challenge:** Contextualizing different camera angles and environments.
* **Solution:** The Ollama inference payload now includes the `node_id`. The prompt dynamically adjusts: *"You are reviewing a frame from 'Front_Gate_Camera'. Describe any human activity."* The VLM processes the queue sequentially using the local GPU/NPU.

### Component D: Vector Storage & RAG Interface
* **Role:** The semantic memory layer.
* **Solution:** The existing `sqlite-vec` architecture remains untouched. The schema is simply updated to include `source_type` (Screen vs. IoT) and `node_id`. This allows hybrid queries: *"Show me what I was coding on my screen when the delivery guy arrived at the front door."*

## 4. Financial Analysis & Feasibility

This distributed approach maximizes the economic advantage of the Local-First architecture.

* **Hardware Cost Asymmetry:** Instead of buying proprietary $200 "AI Cameras" from big tech companies, users buy $5 bare-metal ESP32 components. The "AI" is retroactively applied by the user's existing PC hardware.
* **The Cloud Trap:** Streaming 4K video feeds from 5 cameras to a cloud server for continuous AI analysis requires massive upload bandwidth and would cost hundreds of dollars a month in cloud compute. 
* **The Local/Edge Model:** Bandwidth stays strictly on the Local Area Network (LAN). Cloud OpEx remains at zero.

## 5. MVP Execution Plan (Step 1)

To validate the distributed thesis, the MVP expands the existing Chronos daemon to accept external network payloads.

**Scope for Version 0.2 (IoT Hub Proof of Concept):**
* **Core Language:** Rust (Tokio/Axum).
* **Deliverable:** 1. A new `chronos-receiver` crate that exposes a `POST /api/v1/frame` endpoint.
    2. A simple C++/Arduino script for an ESP32-CAM to capture and send a photo every 10 seconds to that endpoint.
    3. The Chronos daemon processes the ESP32 frame through the existing Ollama integration and saves the result to SQLite, tagged with `source: esp32_cam_01`.
    4. Terminal CLI is updated to support `--source` filtering (e.g., `just run chronos-daemon query --source esp32_cam_01`).
