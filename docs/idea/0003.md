# Executive Proposal & Technical RFC: Project "Chronos Vanguard"
**The Air-Gapped Enterprise Semantic VMS (Video Management System)**

## 1. Executive Summary

Project Chronos Vanguard represents the ultimate scalability milestone for the Chronos architecture. Evolving from a single-node context engine and a lightweight IoT hub, Vanguard transforms the system into a professional, 24/7 Enterprise Video Management System (VMS). It is engineered to ingest continuous RTSP (Real-Time Streaming Protocol) feeds from dozens of high-definition ONVIF IP cameras or legacy DVRs/NVRs.

**The Business Differentiator:** Traditional enterprise VMS solutions (e.g., Verkada, Avigilon) lock customers into expensive, recurring cloud storage contracts and expose highly sensitive corporate or industrial footage to third-party datacenters. Chronos Vanguard offers a **100% Zero-Cloud, On-Premise AI Appliance**. By leveraging local multi-GPU clusters and a highly optimized "Inference Cascade," it provides state-of-the-art semantic search, behavioral anomaly detection, and automated logging without a single frame ever leaving the client's Local Area Network (LAN).

* **For Marketing:** *"Military-grade semantic surveillance. Complete situational awareness, completely offline."*
* **For Finance:** Shift from an unsustainable monthly OpEx (Cloud AI inference and bandwidth) to a single, predictable CapEx (Local hardware appliance), achieving infrastructure ROI within months.

## 2. Product Vision & Go-to-Market

### The Problem
Modern command centers are drowning in digital noise. Security operators stare at walls of 32 monitors, inevitably suffering from visual fatigue. When an incident occurs, investigators waste days manually scrubbing through "dumb" video timelines. Existing "smart" cameras trigger thousands of false positives due to rain, shadows, or insects, leading to severe alert fatigue.

### The Solution (User Journey)
1. **Continuous Ingestion:** Chronos Vanguard connects directly to existing IP camera streams over the local network, acting as an NVR (Network Video Recorder).
2. **The Inference Cascade:** Instead of analyzing every frame with a heavy Vision Language Model (VLM), the system uses a tiered filtering approach (Motion -> Object -> Semantic) to discard useless frames.
3. **Instant Retrieval:** An investigator opens the terminal or web dashboard and types: *"Show me all instances of a white delivery van entering the loading dock after 8:00 PM last month."* The local database returns the exact timestamps, semantic logs, and video snippets instantly.

### Enterprise Ecosystem Use Cases
* **Critical Infrastructure (Air-Gapped):** Power plants, military bases, and data centers that are legally forbidden from connecting their internal camera networks to the public internet.
* **Retail Shrinkage Prevention:** Passively monitoring checkout lanes. The VLM logs: *"Customer placed item in bag without scanning,"* instantly indexing the event for loss prevention teams.
* **Warehouse Safety:** Automated 24/7 compliance logging: *"Forklift operating at unsafe speed in Sector B"* or *"Worker entered restricted zone without a high-visibility vest."*

## 3. Detailed Technical Architecture

To process 32+ simultaneous 1080p video streams without melting the hardware, Vanguard requires a sophisticated concurrency model and a multi-tier AI pipeline.

### Component A: The RTSP Ingestion Pipeline (Rust/GStreamer)
* **Role:** The video collector.
* **Technical Challenge:** Decoding H.264/H.265 video streams from multiple cameras requires heavy CPU/GPU decoding resources.
* **Solution:** A new Rust crate (`chronos-rtsp`) utilizing FFI bindings to GStreamer or FFmpeg. The pipeline connects to the RTSP streams and extracts a single keyframe (I-frame) every 1 to 2 seconds per camera, dropping the rest of the video data to save memory, placing the extracted frames into a highly concurrent ring buffer.

### Component B: The Inference Cascade (The GPU Funnel)
* **Role:** The filtration system. This is the core IP of Vanguard. You cannot run a 32-camera feed directly into Moondream/Ollama.
* **Tier 1 - Motion Detection (CPU):** A blazingly fast pixel-difference algorithm. If the frame is identical to the last one (e.g., an empty hallway), it is immediately dropped. Cost: 0 GPU cycles.
* **Tier 2 - Object Detection (GPU/TensorRT):** Using a lightweight ONNX model like YOLOv8. It checks the frame for specific classes (Person, Car, Truck). If the frame only contains a swaying tree, it is dropped. Cost: ~10ms per frame.
* **Tier 3 - Semantic VLM (GPU/Ollama):** Only if Tier 1 and Tier 2 are triggered does the frame get sent to the Vision Language Model. The VLM generates the rich text description. Cost: ~1-2 seconds per frame.

### Component C: Multi-GPU Worker Pool
* **Role:** Hardware utilization.
* **Solution:** The `chronos-daemon` is updated to act as a load balancer. If the appliance has dual RTX 4090s or multiple edge NPUs, the Rust executor distributes the Tier 3 VLM tasks across the available hardware asynchronously, ensuring no single GPU becomes a bottleneck.

### Component D: Vector Storage & Event Archiving
* **Role:** The immutable audit log.
* **Solution:** The `sqlite-vec` database stores the semantic text and embeddings. Additionally, a local storage manager compresses the triggered frames and saves them to a local NVMe RAID array, linking the image filepath to the SQLite event ID for quick visual retrieval.

## 4. Financial Analysis & Hardware Economics

The "Appliance" model is the key to enterprise sales. 

* **The Competitor Model (Cloud VMS):** $200 per camera + $20/month per camera for cloud AI licensing. For a 32-camera warehouse, that is $6,400 upfront + $7,680 per year in recurring OpEx forever.
* **The Vanguard Model (Edge Appliance):** The client keeps their existing $50 ONVIF cameras. They purchase the "Chronos Vanguard Server" (a 2U rack server with an AMD EPYC CPU and dual enterprise GPUs) for a one-time CapEx of $8,000 to $12,000. 
* **The Result:** Total data sovereignty, sub-second latency, and the hardware pays for itself in less than 18 months compared to cloud solutions.

## 5. MVP Execution Plan (Step 3)

**Scope for Version 0.3 (Local RTSP & Cascade Proof of Concept):**
* **Core Language:** Rust.
* **Deliverable:** 1. Implement the `chronos-rtsp` crate to successfully connect to a single IP camera or a test video file and extract 1 frame per second.
    2. Integrate the `ort` (ONNX Runtime) crate to run a local YOLOv8 model as Tier 2 of the Inference Cascade.
    3. Wire the output of the YOLO detection to the existing Ollama VLM integration.
    4. Provide a CLI command to start the VMS mode: `just run chronos-daemon vms --config cameras.toml`.
